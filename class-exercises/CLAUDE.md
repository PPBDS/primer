# CLAUDE.md — Primer class exercises (`class-exercises/`)

You are authoring a **Primer class exercise** — one of three artifacts the project can build from a single problem, alongside the book chapter (`book/`) and the learnr tutorial (`primer.tutorials/`). Start at the repo index [`../CLAUDE.md`](../CLAUDE.md) for the curriculum and the collaboration protocol.

**The base tutorial guide does not govern class exercises.** That guide (in PPBDS/ai-rules) is the contract for tutorials only; its core workflow (AI-mediated `analysis.qmd`, learnr question types, CP/CR evidence) does not transfer to the classroom. A class exercise is governed by this file plus the Primer's shared guide parts ([`../guide/guidance.md`](../guide/guidance.md), [`../guide/tables.md`](../guide/tables.md), [`../guide/concepts-and-drops.md`](../guide/concepts-and-drops.md), [`../guide/curriculum.md`](../guide/curriculum.md)) — do not read the base guide in, and do not track overrides against it.

> **Reference example: [`resumes/`](resumes/).** The format below is established; `resumes/` is the canonical worked example.

## One seed, up to three renderings

The load-bearing idea: **the upstream material is shared, and the artifact is downstream.** Given a dataset, a Quantity of Interest (QoI), a model, and the Preceptor/Population Tables, you can render a **chapter**, a **tutorial**, or a **class exercise** — or any subset. **We do not always build all three.** A problem may have a tutorial but no class exercise, a chapter but no tutorial, and so on. In practice, **most class exercises will not have a counterpart chapter or tutorial** — the standalone exercise is the common case. **No class exercises will be built for curriculum problems whose chapter and tutorial already exist.** An exercise may render a curriculum slot whose other artifacts are still unbuilt (e.g. `governors`, slot 14 — update that numbered seed) or a brand-new standalone problem (e.g. `resumes` — unnumbered seed). (An earlier `recruits` exercise, paired with the built chapter/tutorial, was retired in July 2026.)

That shared upstream has a home whether or not the other artifacts exist: the **seed spec** at [`../guide/per-problem/<id>.md`](../guide/per-problem/). A standalone class-exercise problem gets an **unnumbered** seed file there (e.g. `hotels.md`), with a Renderings line of `class exercise` only and its EMH tier and predictive/causal framing declared explicitly. **David normally proposes the problem** for a new standalone exercise; occasionally he will ask Claude to suggest candidates instead. Either way, write the seed (with David, per the collaboration protocol) before drafting the exercise. Each seed names the dataset, the "Imagine that you are…" scenario, the broad question and the narrow QoI, the causal/predictive framing, the data-prep step, the final model, the Preceptor and Population Table column structure, and a **Renderings** line tracking which of the three artifacts exist or are planned. A class exercise **consumes that seed; it does not re-derive it.** The question and the model are the same question and model the chapter and tutorial use.

So this guide is **not** the place to specify data, question, or model — those live in the seed. This guide specifies only what is *distinctive about rendering a seed as a class exercise*: the in-class format, the file layout, and how it differs from the tutorial built from the same seed.

## How a class exercise is built

**The conventions below are defaults, not a straitjacket.** They were originally accumulated against a single Easy-tier predictive example (the since-retired `recruits`). Class exercises will vary in complexity and therefore in the sorts of questions they ask; expect this guide to grow *less* prescriptive over time as more exercises exist. Where a rule is clearly Easy-tier-specific (e.g. which `marginaleffects` functions to limit to), read it as the Easy-tier default, not a universal requirement.

A class exercise is a folder of Quarto documents named for the problem — **no number** (the curriculum number is irrelevant in the classroom). See [`resumes/`](resumes/) for the worked example.

- **One exercise per problem.** A single exercise (one answer key, one folder) covers the full virtue arc. The *student* documents are split by class meeting (below), but they are renderings of the one exercise, not separate exercises.
- **Sized for a week, worked in small groups.** An exercise anchors a week of class meetings, not a single sitting. The default split for a three-meeting week: **Wisdom + Justice** in one meeting, **Courage** in its own, **Temperance** in its own. A two-meeting week does Wisdom/Justice on day 1 and Courage/Temperance on day 2. Size each virtue section so those pairings fit their meeting. Students work in pairs or small groups during class, with the day's document doubling as the discussion record.

- **Folder.** `class-exercises/<name>/`, where `<name>` is the problem's main name (e.g. `resumes`). Not numbered.
- **Four QMDs, one exercise.** The **answer key** is the master document; the three **per-meeting student files** are assembled from it:
  - `<name>-answers.qmd` — the **TF answer key**: every question with our answers filled in, plus the worked Preceptor/Population Tables, covering the whole arc. The title must read as the answer version (e.g. "Resumes Answer Guideline").
  - `<name>-wisdom.qmd` — day 1: the Wisdom and Justice **questions**, blank, for students to type answers into.
  - `<name>-courage.qmd` — day 2: opens with **our answers** to all the Wisdom and Justice prompts (copied verbatim from the answer key, worked tables included), then the Courage questions, blank.
  - `<name>-temperance.qmd` — day 3: opens with our answers to Wisdom, Justice, **and** Courage, then the Temperance questions, blank.
  - Each student file begins with the same setup chunk and the same `## The Question` section, so every day stands alone; each later day gives students the benefit of correct answers to everything that has gone before. In courage/temperance, one orientation sentence after `## The Question` says which sections are answers and what today's work is. A side benefit: the embedded Courage answers define the fitted model object, so the Temperance file's `marginaleffects` questions run against it directly.
  - Every question appears, identically worded, in the answer key and in exactly one student file; embedded answer sections are verbatim copies from the key.
  - **The underlying paper, when there is one** (as with `governors` or `resumes`), is included in the exercise folder as `<name>-paper.pdf` and ships in the class repo students fork. List it in the README's file list. (The folder `.gitignore` ignores `*.pdf` render output but negates `!*-paper.pdf`.) `primer.data`'s `inst/papers/` is the first place to look for the PDF.
  - A third file, `README.md`, is written at authoring time and lives in the exercise folder: it is the landing page for the **class repo** (see *Setting up for class* below), so it describes that repo's layout — the three per-meeting student QMDs (with their day mapping and the cumulative-answers design), the answer key, and `data/<name>.csv`. Contents: the scenario sentence and the question, the file list, fork/clone instructions, required software and packages (**including `gt`**: students don't build Preceptor/Population Tables, but rendering the courage/temperance files — which embed the worked tables as answers — requires it), and how to render. In the clone step, say "Open the folder in your IDE. Most of our students use VS Code." — do not name RStudio or Positron. See [`resumes/README.md`](resumes/README.md) for the model.
- **Quarto, `echo: false`, dual format.** Every file is a `.qmd` with a YAML `execute: echo: false` block and a `setup` chunk loading the analysis packages (`tidyverse`, `tidymodels`, `marginaleffects`). The answer key's setup chunk additionally loads `gt` and `broom`; the **courage and temperance student files reuse the answer key's setup verbatim**, since they execute embedded answer chunks (including the `gt` tables); the wisdom file uses the lean setup. **The data ships in the exercise folder as `data/<name>.csv`** (exported from its source package if necessary), and **every file must handle the data identically** — all read the CSV in the same setup code. This makes the primer copy and the class-repo copy identical. Student files render **HTML only** (`format: { html: default }`); only the answer key renders to both formats, with `html: { embed-resources: true }` + `pdf: default`, because **each exercise directory keeps the rendered answer key** — `<name>-answers.html` (self-contained, so no `_files/` dependency) and `<name>-answers.pdf` are committed and ship in the class repo (the folder `.gitignore` negates them). Re-render both after any edit to the answer key. All other render output (`.html`, `.pdf`, `_files/`) stays git-ignored — do not commit it. (PDF caveat: Quarto converts the `gt` HTML tables to LaTeX tables cleanly, but the diagonal hatch on unobservable potential outcomes does not survive; the footnotes still explain it.)
  - **Per-chunk echo overrides, in the answer key only.** Override the document's global `echo: false` to `#| echo: true` on every chunk that shows pedagogically meaningful code: EDA plots, candidate-model fits, the chosen model, model-diagnostic chunks, and the `marginaleffects` workflow (`predictions()`, `avg_predictions()`, `avg_comparisons()`, `plot_predictions()`). Leave the global `echo: false` default in place only for chunks that build the Preceptor/Population `gt` tables — students and TFs should see the modeling and plotting code, but not the table-construction machinery.
- **Structure (mirrors the example).** **Every QMD** begins with a `## The Question` section holding the two scenarios and the two Preceptor's Questions (the answer key's version opens with one or two additional sentences of data context from the seed), followed by the virtue arc Wisdom → Justice → Courage → Temperance.
- **Virtue sub-sections (required, all files).** Each virtue's key components appear as `###` sub-sections, identical wherever the virtue appears:
  - **Wisdom:** `### Preceptor Table` (the units/outcomes/covariates bullets — they *are* the table's components — followed by the describe-the-table prompt, and in the answer key the worked tables) · `### Exploratory Data Analysis`
  - **Justice:** `### Population Table` · `### Assumptions` · `### Probability Family and Link Function`
  - **Courage:** `### Candidate Models` · `### The Data Generating Mechanism` · `### Model Checking`
  - **Temperance:** `### Predictions` · `### Comparisons` · `### Visualizations` · `### Interpretation` · `### Humility` **Two "Imagine that…" scenarios (required).** The student file always includes **two** "Imagine that…" scenarios, given to students verbatim — never ask students to invent one; experience shows that is too hard. One scenario is **causal** and one is **predictive**, each set at a different moment in time with a different set of units. Tell students to focus on just one of the two scenarios (do not assume breakout rooms or assign scenarios) and answer every conceptual prompt from that person's point of view; the same data and the same fitted model serve both scenarios. The handout wording is: "There are two scenarios. Focus on just one of them:" Within one scenario's answers, do not mix paradigms — predictive language for the predictive scenario; potential-outcomes and treatment language only for the causal one. The answer key answers each conceptual prompt twice, labeled by scenario, **with the two scenarios' answers in separate paragraphs** so they are easy to compare and contrast, and builds the Preceptor and Population Tables for both. **Do not include random headers or running meta-commentary transitioning between sections (e.g., remove transitional phrases like "Now that we know the question..." or "Before we fit a model...").** After every question, include 3 blank lines so students have space to write answers. In the answers document, include an empty line before and after the answer or code cell.
- **Student handout philosophy.** The student file should function like a guided worksheet: short prompts, reasoning questions, and a few light data-exploration tasks. It should not spend students' time copying code, debugging, or formatting tables. **Omit extended "Imagine that you are..." narrative, dataset background, or questions asking what question the analyst might try to answer** — but keep the single sentence that states the scenario, since it's needed to pose the Preceptor's Question for Class (see below). The instructor file should contain the missing code, completed written answers, completed tables, rendered figures, and only the comments that add pedagogical value.
- **Answer-key fidelity.** The answer key should preserve the student files' headings, section order, question order, and wording as closely as possible. The only acceptable differences are the filled-in answers, the worked code, the rendered tables, and the rendered figures. **Fidelity is about wording and order, not length.** The answer key may run much longer than the handout — extra plots, computed asides, and discussion prompts are fine there. Don't compress the answer key to match the handout's brevity, and don't pad the handout to match the answer key's length.
- **Prompt design.** Questions should be framed as direct analytical prompts, not as prompts that reveal the answer. Avoid wording such as "Why is a linear model reasonable?" when a more neutral prompt such as "Why is this model reasonable?" is better.
- **Preceptor's Question for Class (required).** In the Wisdom section, begin by stating one focused question **per scenario** — predictive for the predictive scenario, causal for the causal one — targeting each scenario's own units and moment in time directly (e.g., "What is the average causal effect of having a black-sounding name, rather than a white-sounding name, on the probability that a job applicant in Chicago today receives a callback?") instead of analyzing current sample differences. These questions anchor the class's week of work. **Do not follow them with "Is your scenario's question causal or predictive?"** — the question is too easy, since the causal scenario's question says "causal effect" explicitly.
- **End-of-exercise summary.** Students should answer the reflection prompts by briefly identifying the modeling strategy used (what was modeled and how predictions were obtained) and the main remaining source of uncertainty in the results. Responses should emphasize that conclusions are predictive and depend on model assumptions, data representativeness, and sampling variability. Model after the questions asked in `temperance` from the Primer.
- **Fill-in bullet prompts.** When the exercise asks students to identify the units, outcome, covariates, or quantity of interest, include a small bullet list. **In both files, provide exactly three blank lines below the prompt element for the student to record their answer. In the answer key, place the filled-in answer text specifically on the second line of those three blank lines.**
- **Table Description Prompts.** When prompting students to conceptualize tables, use the following standardized text formats:
  - For the Preceptor Table: **"Describe the key components of a Preceptor Table for this problem; Use words like 'units,' 'outcomes,' and 'covariates.'"** The corresponding answer text should state, using the problem's actual column names: "The Preceptor Table should have one row per [unit]. In this exercise, the columns are [unit name], [outcome name], and [covariate name(s)]." **Do not include logistical meta-text indicating that the instructor will display the table after discussion.**
  - For the Population Table: **"Describe the key components of a Population Table for this problem."**
- **Assumptions and Counter-examples (Justice).** In the Justice section, do not ask generalized questions about realistic violations of assumptions. Instead, explicitly list specific blank fields for all core assumptions using this exact format:
  - **"Counter-example to the assumption of validity:"**
  - **"Counter-example to the assumption of stability:"**
  - **"Counter-example to the assumption of representativeness:"**
  - **"Counter-example to the assumption of unconfoundedness:"**
- **Exploratory Data Analysis (EDA) Exercises (required).** Within the Wisdom section in both files, insert 2–3 short exercises that ask students to explore the data. Each exercise: (1) is written as "Create a [plot/table/calculation] that [specific task]" — **and nothing more: no "Briefly describe what it shows" or other open-ended follow-up.** Making the artifact is the whole student task; open-ended describe-it questions are too hard. The answer key carries the noticing prose, which TFs use for discussion and which students receive at the end of the week; (2) surfaces a key feature of the outcome's relationship to the main covariate(s); (3) in the student file, states only the prompt with no code; (4) in the answer key, includes sample code (`{r}` chunk, `echo: true`), sample output, a note on what students should notice, and optionally a follow-up question for the TF to pose in discussion; (5) focuses on natural questions ("What does the distribution look like?", "How do these groups differ?", "Is there a relationship?", "Why might sample size matter?") tied to concepts that matter for the main question. Together, 2–3 exercises give students a visceral sense of the data before modeling, making the model's assumptions and choice transparent.
- **Plot conventions (answer key).** Every plot needs an informative `subtitle`, and **the code students see must be simple**: no helper computations whose only purpose is to feed a label — write the subtitle as plain hardcoded text. Prefer qualitative claims; when a subtitle states a number, verify it against computed output before writing it in. Keep only the computations the plot itself needs. `theme_minimal()` is the house theme. Show the code for plot-producing chunks (`#| echo: true`); keep table-construction code hidden. Below each rendered plot, add one to two sentences of prose (outside the chunk) stating what it shows and the insight it gives — don't leave a plot to speak for itself. Total plots across an exercise should run 2–3 per EDA section, plus the model-diagnostic and prediction plots in Courage/Temperance.
- **Preceptor and Population Tables use the project standard — never improvised.** Build them with the full `gt` pipeline from [`../guide/tables.md`](../guide/tables.md) §10: spanners with fixed IDs (`unit_span`, `outcome_span`, `treatment_span`, `covariates_span`), the footnote set, the hatch helper for causal counterfactuals, the 11-row Population Table layout, and the `opt_css` → `as_raw_html()` → `cat()`-in-`results: asis` ending. The fastest correct path is to **copy the matching tables from the problem's chapter** (`../book/<NN-name>.qmd`) or tutorial and adapt them. Do **not** write ad-hoc flat `gt()` tables; the answer key's tables should look like the chapter's. **Additionally, every class-exercise table needs an explicit `gt::cols_width()`** (roughly 80–170px per column, sized to content) placed after the spanners: in the class-exercise rendering context the shrink-wrap pipeline otherwise collapses the table to minimum width and cell text wraps, which is what made early tables look far worse than the chapters'.
- **Seed-driven content.** Dataset, scenarios, outcome, covariates, treatment, model, and table column structure all come from the seed; do not re-derive them. After building, set the seed's **Renderings** line `class exercise ✓`.
- Multiple complementary visualizations per EDA prompt. Any prompt asking students to "create a plot" examining a relationship should be answered, in the answer key, with 2–3 plots that each surface a different feature of the data (e.g., boxplot for medians/spread, violin+jitter for sample-size visibility, density for distributional shape) rather than 2–3 near-duplicates of the same chart type.
- Name every code chunk descriptively. Chunk labels should state purpose, not position (`eda-height-sex-boxplot`, not `unnamed-chunk-3`; `chosen-model`, not `fit`). This makes show_file()-style CP/CR workflows and later cross-referencing tractable.
- Follow the Question → Wisdom (with EDA) → Justice → Courage → Temperance sequence. Remaining uncertainty + bias sources close every exercise.
- Candidate models must be motivated, not just listed. Each candidate model chunk should be followed by one sentence explaining what the result tells us and how it informs the next candidate (fit the binary covariate alone, then the continuous covariate alone, then explain why combining doesn't change the first coefficient, then justify the final pick on parsimony grounds).
- Always include at least one model-diagnostic chunk (fitted-vs-actual distribution comparison, or a residual plot) between the chosen model and Temperance, even in an Easy-tier exercise — and a corresponding student-facing prompt asking them to produce it. This is what makes "Courage" feel like genuine model-checking rather than a single `fit()` call.
- **Marginal effects workflow (Temperance section — mandatory).** The `marginaleffects` package is the single bridge from parameters to outcome-scale answers. Use this function sequence **consistently in this order across every Temperance section:**
  1. **`predictions()`** — establish that the model produces one prediction per unit; mostly a stepping stone shown briefly to anchor what a prediction is. Skip verbosity; show just enough to see the structure.
  2. **`avg_predictions(by = ...)`** — the actual headline numbers students must remember: group means with confidence intervals. This is the workhorse function for nearly every predictive Easy/Medium tier exercise.
  3. **`avg_comparisons(variables = ...)`** — convert "two group means" into "one difference with one CI," which is almost always the literal quantity the question asks for and avoids students eyeballing overlapping intervals.
  4. **`plot_predictions()` / `plot_comparisons()`** — visual counterparts, used once each (not redundantly). Pair with prose: one to two sentences stating what the plot shows and the insight it conveys.
  5. **`slopes()` / `plot_slopes()`** — reserve for continuous-covariate questions only (e.g., a Medium-tier problem where the QoI is a rate of change), since slopes are confusing applied to a 0/1 binary covariate.
  - **Easy/Medium tier:** Limit to `avg_predictions()` + `avg_comparisons()` + one plot of each. Do not introduce students to `comparisons()` (the non-averaged version) until the point of the exercise is heterogeneous effects across subgroups.
  - **Medium/Hard tier:** Introduce `comparisons()` only when examining effect heterogeneity becomes the pedagogical goal; otherwise follow the Easy/Medium pattern.
  - Each function call gets one sentence of plain-language interpretation: "what this estimates and why we care." Do not leave output to speak for itself.
- Use realistic, mechanism-specific counterexamples for assumptions, not generic templates. "The slope might differ because X mechanism changed" beats "the relationship might be different over time." Reference the book chapters.
- Distinguish practical significance from statistical significance explicitly. After reporting a confidence interval excluding zero, add one sentence translating the magnitude into a real-world consequence (uniform sizing, budget impact, etc.).
- Quantities of interest beyond the headline number (max, percentiles, distribution of a sample statistic) should be named even when not computed in full, to model the habit of asking "what else does the [decision-maker] need?"

## Setting up for class

**One class repo per class meeting.** Each repo is named for the base name of the main file within it — `governors-wisdom`, `governors-courage`, `governors-temperance` — never for the whole exercise. When David has checked the files and says **"set up for class"** for a given meeting:

1. **Create a new public repo on David's GitHub account** (`davidkane9`), named for that meeting's student file (e.g. `governors-wisdom`). If the name is already taken on his account, ask before picking an alternative.
2. **Contents — exactly these, nothing else:**
   - That meeting's **student QMD** (e.g. `governors-wisdom.qmd`).
   - The **data**, as `data/<name>.csv` — already what the QMD reads, so no setup-chunk changes are needed at deploy time.
   - **The underlying paper** (`<name>-paper.pdf`), when the exercise has one — pushed every day.
   - The exercise folder's **`README.md`**, copied as-is — it is written at authoring time (see *How a class exercise is built*) to read correctly in any of the exercise's per-meeting repos.
3. **Verify the QMD renders from the CSV** before pushing.
4. **Report the repo URL to David.** In class, he gives that URL to students, who fork the day's repo so each has their own working copy.

The answer key and its rendered HTML/PDF stay in the primer repo — they are not deployed to the per-meeting repos. (Students effectively receive prior-day answers through the cumulative courage/temperance files; the full answer key is shared at the end of the week, at David's discretion.) The primer repo remains the source of truth; class repos are deployment copies. Fixes go here first, then get pushed out.

## What is already fixed

- **Canonical definitions are inherited.** [`../book/key-concepts.qmd`](../book/key-concepts.qmd) is the single source of truth for every definition; the in-class exercises are named there explicitly as one of the artifacts that must agree with it. Never redefine a term — quote or link the canonical wording.
- **Tier and framing.** Class exercises use the same **EMH (Easy/Medium/Hard) tripartite breakdown** as chapters and tutorials. If the problem sits in the 16-slot curriculum, the slot fixes its tier (05–08 Easy, 09–12 Medium, 13–16 Hard) and predictive/causal framing — see [`../guide/curriculum.md`](../guide/curriculum.md). A standalone exercise (the common case) declares its tier and its predictive/causal framing explicitly.
- **Concepts introduced in order.** Never use a concept the curriculum introduces in a later position (base guide; Primer schedule in `../guide/curriculum.md`).
