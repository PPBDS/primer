# CLAUDE.md — Primer class exercises (`class-exercises/`)

You are authoring a **Primer class exercise** — one of three artifacts the project can build from a single problem, alongside the book chapter (`book/`) and the learnr tutorial (`primer.tutorials/`). Start at the repo index [`../CLAUDE.md`](../CLAUDE.md) for the curriculum, the base-guide relationship, and the collaboration protocol.

> **Reference example: [`recruits/`](recruits/).** The format below is established; `recruits/` is the canonical worked example. A few rendering details (listed under **Open questions**) are still unsettled — ask before inventing structure for those.

## One seed, up to three renderings

The load-bearing idea: **the upstream material is shared, and the artifact is downstream.** Given a dataset, a Quantity of Interest (QoI), a model, and the Preceptor/Population Tables, you can render a **chapter**, a **tutorial**, or a **class exercise** — or any subset. **We do not always build all three.** A problem may have a tutorial but no class exercise, a chapter but no tutorial, and so on.

That shared upstream already exists and already has a home: the **seed spec** at [`../guide/per-problem/<id>.md`](../guide/per-problem/). Each seed names the dataset, the "Imagine that you are…" scenario, the broad question and the narrow QoI, the causal/predictive framing, the data-prep step, the final model, the Preceptor and Population Table column structure, and a **Renderings** line tracking which of the three artifacts exist or are planned. A class exercise **consumes that seed; it does not re-derive it.** The question and the model are the same question and model the chapter and tutorial use.

So this guide is **not** the place to specify data, question, or model — those live in the seed. This guide specifies only what is *distinctive about rendering a seed as a class exercise*: the in-class format, the file layout, and how it differs from the tutorial built from the same seed.

## How a class exercise is built

A class exercise is a pair of Quarto documents in a folder named for the problem — **no number** (the curriculum number is irrelevant in the classroom). See [`recruits/`](recruits/) for the worked example.

- **Folder.** `class-exercises/<name>/`, where `<name>` is the problem's main name (e.g. `recruits`). Not numbered.
- **Two files, one exercise:**
  - `<name>.qmd` — the **student handout**: questions and prompts the student answers by typing directly into the document.
  - `<name>-answers.qmd` — the **TF answer key**: the *same* questions with our answers filled in, plus the worked Preceptor/Population Tables. Title carries `(Answer Key)`.
  - The student file is the answer key with the answer prose and the worked `gt` tables removed — building those is the student's task. Keep the two in sync: every question in one appears, identically worded, in the other.
- **Quarto, `echo: false`.** Both files are `.qmd` with a YAML `execute: echo: false` block and a `setup` chunk loading the packages the data needs (`tidyverse`, `gt`, and whichever package ships the dataset — `primer.tutorials` for `recruits`, `primer.data` for most others). Render with `quarto render` to confirm they build; the generated `.html`/`_files` are git-ignored, do not commit them.
- **Structure (mirrors the example).** Each handout should follow the virtue arc Wisdom → Justice → Courage → Temperance. Each exercise should commit to exactly one modeling paradigm. Predictive exercises should focus on units, outcomes, covariates, quantities of interest, assumptions for inference, and the fitted model; causal exercises should focus on treatment, potential outcomes, and the assumptions that justify causal claims. Do not mix predictive and causal reasoning in one exercise.
- **Student handout philosophy.** The student file should function like a guided worksheet: short prompts, reasoning questions, discussion starters, and a few light data-exploration tasks. It should not spend students' time copying code, debugging, or formatting tables. For class exercises that culminate in a final product, the student handout should use only a few very basic prompts and leave the final deliverable workflow to the later sections. The instructor file should contain the missing code, completed written answers, completed tables, rendered figures, and only the comments that add pedagogical value.
- **Answer-key fidelity.** The answer file should preserve the student file's headings, section order, question order, and wording as closely as possible. The only acceptable differences are the filled-in answers, the worked code, the rendered tables, and the rendered figures.
- **Scenario framing.** Every class exercise should begin with a short “Imagine…” section that places students in a realistic role and a short explanation of the data, including what each row represents and which variables are available. The scenario should be grounded in the seed and should not turn into a long setup block.
- **Prompt design.** Questions should be framed as what a student or analyst might ask in that scenario, not as prompts that reveal the answer. Avoid wording such as “Why is a linear model reasonable?” when a more neutral prompt such as “Why is this model reasonable?” is better.
- **Preceptor's Question for Class (required).** In the Wisdom section, immediately after the main scenario question, include a subsection titled "Question for Class This Week." State one focused predictive or causal question that is obvious and natural given the problem's outcome and covariates. This question anchors the class's week of work. In the student handout, ask students to fill in blanks for units, outcome, covariates, treatment (if any), and quantity of interest. In the answer key, provide filled-in answers.
- **Exploratory Data Analysis (EDA) Exercises (required).** Between the Courage section and the Temperance section in both files, insert 2–3 short exercises that ask students to use AI to explore the data. Each exercise: (1) is written as "Use AI to create a [plot/table/calculation] that [specific task]"; (2) asks an exploratory question that surfaces a key feature of the outcome's relationship to the main covariate(s); (3) in the student file, states only the prompt with no code; (4) in the answer key, includes sample code (`{r}` chunk), sample output, a note on what students should notice, and a follow-up question for class discussion plus teaching fellow prompts; (5) focuses on natural questions ("What does the distribution look like?", "How do these groups differ?", "Is there a relationship?", "Why might sample size matter?") tied to concepts that matter for the main question. Together, 2–3 exercises give students visceral sense of the data before modeling, making the model's assumptions and choice transparent.
- **Fill-in bullet prompts.** When the exercise asks students to identify the units, outcome, covariates, or quantity of interest, include a small bullet list with fill-in blanks so students can record the key elements directly.
- **Instructor prompts.** Include occasional prompts directed to the teaching fellow so the exercise can support breakout discussion and help students articulate concepts such as units, outcomes, covariates, quantities of interest, validity, stability, representativeness, Preceptor Tables, Population Tables, and the choice of model. These prompts belong only in the answer-key file, never in the student handout.
- **Final-product workflow.** When a class exercise ends with a publishable artifact, the final section should guide students to create a short Quarto document, include at least one plot, and publish it to GitHub Pages. The instructor answer key should show the completed version of that deliverable. The plot step should explicitly direct students to ask AI to create the plot and to use AI for the coding, rather than relying on manual coding. Before that step, the student handout should include one code block with the preceptor-prepared analysis scaffold that covers the data loading, cleaning, and model fitting steps. That block should be introduced with a short note such as “This is the code Preceptor has prepared for you.” The handout should include optional extension ideas for students who finish early, and those ideas should be specific to the data and model in that exercise rather than generic. They can include prompts for asking AI to inspect the data, identify outliers, suggest better visuals, or propose follow-up questions that strengthen the final report.
- **Preceptor and Population Tables use the project standard — never improvised.** Build them with the full `gt` pipeline from [`../guide/tables.md`](../guide/tables.md) §10: spanners with fixed IDs (`unit_span`, `outcome_span`, `treatment_span`, `covariates_span`), the footnote set, the hatch helper for causal counterfactuals, the 11-row Population Table layout, and the `opt_css` → `as_raw_html()` → `cat()`-in-`results: asis` ending. The fastest correct path is to **copy the matching tables from the problem's chapter** (`../book/<NN-name>.qmd`) or tutorial and adapt them. Do **not** write ad-hoc flat `gt()` tables; the answer key's tables should look like the chapter's.
- **Seed-driven content.** Dataset, scenarios, outcome, covariates, treatment, model, and table column structure all come from the seed; do not re-derive them. After building, set the seed's **Renderings** line `class exercise ✓`.

## What is already fixed

- **Canonical definitions are inherited.** [`../book/key-concepts.qmd`](../book/key-concepts.qmd) is the single source of truth for every definition; the in-class exercises are named there explicitly as one of the artifacts that must agree with it. Never redefine a term — quote or link the canonical wording.
- **The slot fixes tier and framing.** A class exercise sits at its problem's curriculum position, so it inherits that slot's **EMH tier** (05–08 Easy, 09–12 Medium, 13–16 Hard) and its **predictive/causal framing** from the seed. See [`../guide/curriculum.md`](../guide/curriculum.md).
- **Concepts introduced in order.** Never use a concept the curriculum introduces in a later position (base guide; Primer schedule in `../guide/curriculum.md`).

## Open questions (still unsettled — ask David)

The upstream (data / question / model / tables) comes from the seed, and the format and file layout are settled (above). What remains:

1. **One exercise per problem, or one per virtue?** `recruits/` is a single end-to-end exercise. The example title pattern ("Wisdom and Recruits") could instead imply a `wisdom`/`justice`/`courage`/`temperance` file set per problem. Current default: one per problem.
2. **Base-guide inheritance.** Does the base tutorial guide (which governs tutorials) also govern class exercises, govern them with explicit overrides, or not apply? Note the class exercise already departs from the base guide's AI-mediated `analysis.qmd` workflow — students type answers directly into the handout `.qmd` — so at minimum that is an override.
3. **Length / time box and solo-vs-group.** What class duration is one exercise sized for, and is it individual, paired, or whole-class?
4. **Coverage.** Which problems get a class exercise? (Not necessarily all 16 — see the seed **Renderings** lines.)
5. **Guide home.** Keep this guidance here, or move it to a new part under [`../guide/`](../guide/) (e.g. `guide/class-exercises.md`) with this file as a thin router — mirroring how `book/` and `primer.tutorials/` route into `guide/`?

As David settles these, fold the answers in above and trim this list.
